# **AGENTS.md**

# **Market Memory Agent Protocol**

This file defines the operating rules for all AI agents, including Codex, Claude, ChatGPT, and any future automation tools, when modifying or generating content inside the **Market Memory** Obsidian vault.

This vault is a structured financial history knowledge base. It is not a general encyclopedia, not a diary, and not a collection of loose essays.

All agents must read and follow this file before creating, editing, splitting, merging, moving, tagging, linking, or summarizing any note in this vault.

---

# **1. Vault Purpose**

## **1.1 Core Goal**

Market Memory is a long-term knowledge base for studying historical financial, economic, monetary, geopolitical, and market events.

The goal is to build a structured system that helps the user understand:

- how financial crises form
- how asset bubbles develop and collapse
- how debt, currency, banking, and inflation crises spread
- how governments and central banks respond
- how different asset classes behave during different historical regimes
- how past market events can help interpret future risks

The vault should help answer questions such as:

- What happened in a specific decade?
- What historical events resemble the current market environment?
- How did stocks, bonds, real estate, gold, oil, currencies, and crypto behave during similar events?
- What policy responses worked or failed?
- What mechanisms repeatedly appear across different countries and periods?
- What investment lessons can be extracted from historical cases?

## **1.2 What This Vault Is Not**

This vault is not:

- a general history encyclopedia
- a place for vague macro commentary
- a motivational writing space
- a news archive
- a collection of long unstructured essays
- a place for unsourced precise claims
- a country-by-country history textbook

The vault exists to preserve structured, reusable, cross-linked knowledge about financial history and market mechanisms.

---

# **2. Core Principles**

All agents must follow these principles:

1. Prefer mechanism over narration.
2. Prefer facts over opinions.
3. Prefer structure over long prose.
4. Prefer precise language over vague commentary.
5. Separate facts, mechanisms, interpretations, and investment lessons.
6. Avoid emotional, motivational, decorative, or exaggerated language.
7. Every note should serve financial, economic, market, policy, or investment understanding.
8. Every event note must connect to time, pattern, country or region, asset, and concept whenever possible.
9. No orphan notes are allowed unless explicitly marked as draft.
10. Do not create duplicate notes for the same event, concept, country, asset, pattern, or comparison.
11. If information is uncertain, mark it as uncertain.
12. Do not overstate causality.
13. Do not claim that one factor caused an event when the event was multi-causal.
14. Do not delete user-written content unless explicitly instructed.
15. Do not create new top-level folders unless explicitly approved.

Timeline notes should not duplicate full event notes. They should summarize and link to event notes.

- policy response

- investment lesson

# **5. Note Types**

Every note must belong to one of the following types:

```text
event
pattern
country
asset
concept
comparison
timeline
overview
draft
```

## **5.1 Event**

A real historical event.

Examples:

- `1929 - United States - Great Depression.md`
- `1989 - Japan - Asset Bubble Collapse.md`
- `2008 - Global - Global Financial Crisis.md`

## **5.2 Pattern**

A recurring market, economic, or financial pattern.

Examples:

- `Asset Bubble`
- `Debt Crisis`
- `Banking Crisis`

## **5.3 Country**

A country or region financial profile.

Examples:

- `United States`
- `Japan`
- `China`
- `Argentina`

## **5.4 Asset**

An asset-class behavior profile.

Examples:

- `Stocks`
- `Bonds`
- `Gold`
- `Oil`
- `Crypto`

## **5.5 Concept**

A reusable mechanism or financial concept.

Examples:

- `Leverage`
- `Liquidity`
- `Balance Sheet Recession`
- `Currency Mismatch`

## **5.6 Comparison**

A structured comparison between two or more events, countries, assets, patterns, or concepts.

## **5.7 Timeline**

A chronological overview of a decade or period.

## **5.8 Overview**

A landing page inside a folder.

Examples:

- `Asset Bubble Overview.md`
- `Japan Overview.md`
- `Stocks Overview.md`

## **5.9 Draft**

A temporary note that has not been fully structured, tagged, or linked.

Drafts must be clearly marked with:

```yaml
status: draft
```

# **6. File Naming Rules**

## **6.1 General Rules**

File names should be:

- English
- stable
- descriptive
- consistent
- sortable
- compatible with Git, AI tools, and Obsidian

Avoid:

- vague names
- emotional names
- overly long names
- duplicate names
- mixed Chinese-English filenames
- special characters that may cause path issues

## **6.2 Event File Naming**

Use this format:

```text
YYYY - Country or Region - Event Name.md
```

Examples:

```text
1929 - United States - Great Depression.md
1971 - United States - Nixon Shock.md
1973 - Global - First Oil Crisis.md
1985 - Global - Plaza Accord.md
1989 - Japan - Asset Bubble Collapse.md
1992 - United Kingdom - Black Wednesday.md
1997 - Asia - Asian Financial Crisis.md
2000 - United States - Dotcom Bubble Burst.md
2008 - Global - Global Financial Crisis.md
2010 - Europe - European Debt Crisis.md
2020 - Global - COVID Market Crash.md
```

For multi-year events, use the starting year unless the event is historically known by another date.

Examples:

```text
1980s - Latin America - Debt Crisis.md
1990s - Japan - Lost Decade.md
```

## **6.3 Folder Naming**

Folder names should use Title Case English.

Examples:

```text
Asset Bubble/
Real Estate Crash/
United States/
Balance Sheet Recession/
Bubble Comparisons/
```

## **6.4 Headings Inside Notes**

The main title may be bilingual.

Example:

```markdown
# 1989 日本资产泡沫破裂 Japan Asset Bubble Collapse
```

File name should remain English, but note content may use Chinese explanation where useful.

# **7. YAML Metadata Rules**

Every structured note must include YAML frontmatter.

YAML is required because it acts as the database layer of the vault.

## **7.1 General Required Fields**

Every structured note should include:

```yaml
---
type:
status:
title:
tags:
created:
updated:
---
```

Allowed status values:

```text
draft
active
reviewed
archived
```

Definitions:

- `draft`: incomplete, not fully verified, not fully linked
- `active`: usable but may still need improvement
- `reviewed`: checked, structured, linked, and ready for use
- `archived`: old or deprecated, kept for historical reasons

---


## **8.1 Event Field Definitions**

```text
type:
  Must be event.

status:
  draft, active, reviewed, or archived.

title:
  Full note title.

start_year:
  Year the event began.

end_year:
  Year the event ended, if applicable.

decade:
  Example: 1920s, 1970s, 2000s.

region:
  Example: Global, Asia, Europe, North America, Latin America.

countries:
  Countries directly involved.

patterns:
  Recurring patterns connected to the event.

assets:
  Asset classes significantly affected.

concepts:
  Mechanisms needed to understand the event.

impact_level:
  local, regional, global, or systemic.

time_horizon:
  short, medium, long, or structural.

source_status:
  unsourced, partially-sourced, sourced, verified.

sources:
  List of sources if available.

tags:
  Standardized Obsidian tags.

created:
  Creation date.

updated:
  Last update date.
```

# **16. Writing Style Rules**

All content must be written in a concise, analytical, structured style.

## **16.1 Required Style**

Use:

- concise sentences
- bullet points
- clear headings
- factual language
- mechanism-based explanation
- direct comparison
- precise terminology

Avoid:

- long narrative paragraphs
- emotional language
- vague macro commentary
- decorative writing
- motivational language
- unsupported claims
- exaggerated causality
- empty phrases

## **16.2 Forbidden Vague Sentences**

Avoid sentences like:

```text
This crisis had a huge impact on the world economy.
```

Replace with:

```text
The crisis froze interbank lending, widened credit spreads, forced central banks to provide emergency liquidity, and triggered a sharp repricing of risk assets.
```

Avoid:

```text
This was a very important event in financial history.
```

Replace with:

```text
This event changed the monetary regime by ending the dollar's direct convertibility into gold.
```

## **16.3 Separate Facts, Mechanisms, Interpretation, and Lessons**

Do not mix facts, interpretation, and investment lessons in the same section.

Use this structure where possible:

```markdown
## Facts

## Mechanism

## Policy Response

## Asset Price Reaction

## Social and Economic Impact

## Interpretation

## Investment Lessons

## Open Questions
```

# **23. Ingestion Workflow**

When new material is provided, the agent must follow this process.

## **23.1 Step 1: Identify Material Type**

Classify the input as one or more of:

```text
historical event
pattern / mechanism
country / region
asset
concept
comparison
raw note / draft
source excerpt
mixed material
```

## **23.2 Step 2: Extract Atomic Knowledge Units**

Do not store mixed material as one oversized note if it contains multiple independent units.

Example:

If the input discusses Japan’s bubble and includes Plaza Accord, BOJ policy, real estate prices, banking bad loans, and balance sheet recession, the agent should consider splitting it into:

```text
Event: 1985 - Global - Plaza Accord
Event: 1989 - Japan - Asset Bubble Collapse
Concept: Balance Sheet Recession
Pattern: Real Estate Crash
Country: Japan / Lost Decades
Comparison Candidate: Japan 1989 vs US 2008
```

```text
create a new note
update an existing note
split into multiple notes
merge with an existing note
move a note
```

# **26. Source Rules**

The vault does not require a separate source folder by default.

However, every structured note should preserve lightweight source metadata.

## **26.1 Source Status Values**

Use:

```text
unsourced
partially-sourced
sourced
verified
```

Definitions:

```text
unsourced:
  No clear source attached.

partially-sourced:
  Some claims have sources, but not all.

sourced:
  Main factual claims are supported by sources.

verified:
  Important factual claims have been checked against reliable sources.
```

## **26.2 Source Field**

Use:

```yaml
source_status: unsourced
sources: []
```

If sources exist:

```yaml
source_status: sourced
sources:
  - "Book, paper, report, article, dataset, or URL title"
```

## **26.3 Unsourced Claims**

Do not add precise numbers, dates, or causal claims unless they are supported or clearly marked as uncertain.

If the user provides unsourced rough notes, the note should remain:

```yaml
status: draft
source_status: unsourced
```

# **27. Quality Checklist**

Before finalizing any new or edited note, the agent must check:

## **27.1 Structure**

- Does the note have the correct type?
- Is it in the correct folder?
- Does it use the correct template?
- Does it have YAML frontmatter?
- Does it have a clear title?

## **27.2 Metadata**

- Is `type` correct?
- Is `status` correct?
- Are `created` and `updated` present?
- Are tags standardized?
- Are countries, patterns, assets, and concepts listed where applicable?
- Is source status included?

## **27.3 Links**

- Does the note link to related notes?
- Does an event note link to at least one country/region, pattern, asset, and concept?
- Are there any obvious missing links?
- Are there unnecessary or vague links?
- Are placeholder notes marked as draft?

## **27.4 Writing Quality**

- Is the writing concise?
- Are facts separated from interpretation?
- Are mechanisms explained clearly?
- Are investment lessons specific?
- Are vague phrases removed?
- Is uncertainty marked?
- Is causality not overstated?

## **27.5 Duplication**

- Does a similar note already exist?
- Is this note duplicating an existing event, concept, pattern, country, asset, or comparison?
- Should this content update an existing note instead?

## **27.6 Final Review**

Before finishing, the agent should be able to answer:

```text
What type of note is this?
Where does it belong?
What does it link to?
What tags does it use?
What is the main mechanism or lesson?
Is this note useful for future financial or market analysis?
```

# **28. Forbidden Practices**

Agents must not:

1. Create vague notes with no links.
2. Create duplicate notes for the same event or concept.
3. Use inconsistent tags.
4. Mix multiple unrelated events in one note.
5. Write long narrative paragraphs without structure.
6. Add unsourced precise numbers unless a source is provided.
7. Overstate causality.
8. Delete user-written content without instruction.
9. Rename files casually.
10. Create new top-level folders without explicit approval.
11. Use Chinese tags mixed with English tags.
12. Create emotional or motivational summaries.
13. Write generic textbook explanations unless directly needed.
14. Leave important notes orphaned.
15. Treat the vault as a general history archive.
16. Treat current news as verified historical fact without review.
17. Generate investment advice without separating it from historical analysis.
18. Hide uncertainty.
19. Use inconsistent names for the same country, asset, pattern, or concept.
20. Replace structured notes with long essays.

# **31. Final Operating Philosophy**

The vault should follow this principle:

```text
Folders store facts by module.
Links express relationships.
Tags classify notes.
YAML makes the vault queryable.
Comparison notes generate insight.
```

The purpose of Market Memory is not to remember everything.

The purpose is to build a reusable system for recognizing repeated financial patterns across history.

The final standard is:

```text
Every note should be precise, structured, linked, searchable, comparable, and useful for future market reasoning.
```
